JAVA编程思想：二十-注解

Posted on 2019-02-24 Edited on 2020-11-19

注解(也被称为元数据)，为我们在代码中添加信息提供了一种形式化的方法，使我们可以在稍后某个时刻非常方便的使用这些数据

注解使得我们能够以将由编译器来测试和验证的格式，存储有关程序的额外信息，注解可以用来生成描述符文件，甚至或是新的类定义，并且有助于减轻编写“样板”代码的负担

java SE5内置了三种，定义在java.lang中的注解（标准注解）

  • @Override
  • @Deprecated
  • @SuppressWarnings

1.基本语法

注解的定义很像是接口的定义，事实上与其他任何java接口一样，注解也会编译成class文件

在注解中一般会包含某些元素以表示某些值，当分析注解时，程序或工具可以利用这些值，没有元素的注解称为标记注解

有元素注解示例：

在定义注解时，会需要一些元注解(meta-annotation),如@Target和@Retention

  • @Target用来定义你的注解将应用于什么地方（例如一个方法或是一个域）,ElementType可选值包括：

    • CONSTRUCTOR：构造器的声明
    • FIElD：域声明（包括enum元素）
    • LOCAL_VARIABLE：局部变量声明
    • METHOD：方法声明
    • PACKAGE：包声明
    • PARAMETER：参数声明
    • TYPE：类、接口（包含注解类型）或enum声明

    如果想要将该注解应用于所有的ElementType，那么可以省去@Target元注解，不过这不常见

  • @Retention用来定义该注解在哪一个级别可用，

    • SOURCE：在源代码中，注解被编译器丢弃
    • CLASS：类文件中，注解在class文件中可用，但会被jvm丢弃
    • RUNTIME：运行时，JVM在运行期也保留注解，因此可以通过反射机制读取注解信息

  • @Decumented：将次注解包含在javadoc中

  • @Inherited：允许子类继承父类中的注解

大多数情况，程序员自己定义注解，并编写自己的处理器来处理他们

2.编写注解处理器

在使用注解的过程中，很重要的一个部分就是创建与使用注解处理器

一般可使用反射获取注解并对注解进行处理

注解元素的可用类型如下：

  • 所有基本类型（int、float、boolen等）
  • String
  • Class
  • enum
  • Annotation
  • 以上类型的数组

使用其他类型，编译器即将报错（特殊注意的是：也不允许使用任何包装类型）

对于注解的元素：不能有不确定的值，也就是说，元素要么具有默认值，要么在使用注解时提供元素的值

其次对于非基本类型的元素，无论是在源代码中声明时，或是在注解接口中定义默认值时，都不能以null作为其值，这个约束使得处理器很难表现一个元素的存在或是缺失的状态

  • 为类绕开这个约束，我们只能自己定义一些特殊的值，例如空字符串或负数，以表示某个元素不存在。在定义注解的时候，这算是一个习惯用法

如果程序员的注解中定义了名为value的元素，并且在应用该注解的时候，该元素是唯一需要赋值的元素，那么此时无需使用名-值对的这种写法，而只需在括号内给出value元素所需的值即可，这可以应用于任何合法类型的元素，当然这也限制了程序员必须将此元素命名为value,不过这样可能更加简洁

使用对个注解的时候，同一个注解不能重复使用

注解不支持继承

  • 不能使用关键字extends来继承某个@interface

3.使用apt处理注解

apt是注解处理工具

apt与javac一样被设计为操作java源文件

使用apt生成注解处理器时，我们无法利用java的反射机制，因为我们操作的是源代码，而不是编译后的类

4.基于注解的单元测试

在java中，最著名的单元测试就是JUnit,JUnit中就使用了注解

5.总结

注解是java引入的一项非常受欢迎的补充

他提供了一种结构化的，并且具有类型检查能力的新途径，从而使得程序员能够为代码加入元数据，而不会导致代码杂乱且难以阅读，使用注解能够帮助我们避免编写累赘的部署描述文件，以及其他生成的文件，而javadoc中的@deprecated被@Deprecated注解取代的事实也说明，与注释性文字相比，注解绝对更适合用于描述类相关的信息

设计模式总结

什么是设计模式：

  • 某类特定问题的代码设计解决方案
  • 是一套代码设计的经验总结

设计模式的作用

  • 提高代码的复用率，降低开发成本和周期
  • 提高代码的可维护性、可扩展性
  • 使代码更优雅
  • 让代码更容易被他人理解

为什么会有设计模式：因为我们想要编写出可扩展、可复用、灵活、易维护的代码

设计模式的分类

正确的使用设计模式

  • 需求驱动

    设计模式针对软件设计，软件设计针对需求，所以不要为了使用设计模式而使用设计模式，在不合适的场景生搬硬套的使用只会让设计更复杂，软件更难调试和维护

  • 分析成功的模式应用

    对现有的应用实例进行分析和学习，不只是学习设计模式如何实现，更重要是在何种场合使用的设计模式

  • 在编程中领悟模式

    多思考、多实践、灵活应用

  • 避免过度设计

    设计模式是为了让设计简单而不是更复杂，针对一个需求有稳定部分有可变部分，需要判断可能发生变化的程度来确定设计策略和设计风险，过度设计和设计不足同样对项目有害

引用《设计模式解析》书中的一句话：设计模式体现的是一种思想，而思想是指导行为的一切，理解和掌握了设计模式，并不是说记住了23种（或更多）设计场景和解决策略（实际上这也是很重要的一笔财富），实际上接受的是一种思想的熏陶和洗礼，等这种思想融入到了你的思想中后，你就会不自觉的使用这种思想去进行你的设计和开发，这才是最重要的

EffectiveJava(1)-考虑用静态工厂方法代替构造器

Posted on 2019-01-14 Edited on 2020-11-19

第1条：考虑用静态工厂方法代替构造器

静态工厂方法对比构造器的优势：

  • 有名称

  • 不必在每次调用他们的时候都创建一个新对象

  • 可以返回原返回类型的任何子类型的对象

  • 在创建参数化类型实例的时候，它们使代码变得更加简洁,示例：

静态工厂方法的缺点：

  • 主要缺点：类如果不含共有的或者受保护的构造器，就不能被子类化

    不过也有可能不是缺点，因为鼓励使用复合而不是继承

  • 他们与其他的静态方法实际上没有任何区别

    这导致他们没有在API文档中呗明确标识出来，此时通过文档查看如何示例话，比较困难

    此时使用一些静态工厂的惯用名，可以弥补这一劣势，惯用名称：

    • valueOf：该方法返回的实例与他的参数具有相同的值，这样的静态工厂实际上是类型转换方法
    • of：valueOf的简洁替代
    • getInstance：返回的实例是通过方法的参数来描述的，但是不能说与参数具有同样的值，对于Singleton来说，该方法没有参数，并返回唯一实例
    • newInstance：像getInstance一样，但是newInstance能够确保返回的每个实例都与所有其他实例不同
    • getType：像getInstance一样，但是在工厂方法处于不同的类中的时候使用，Type表示工厂方法所返回的对象类型
    • newType：像newInstance一样，但是在工厂方法处于不用的类中的时候使用，Type表示工厂方法所返回的对象类型

总结：简而言之，静态工厂方法和共有构造器都各有用处，我们需要理解他们各自的长处，静态工厂通常更加合适，因此，切记第一反应就是提供公有的构造器，而不先考虑静态工厂。

享元模式(Flyweight Pattern)

定义

运用共享技术有效地支持大量细粒度对象的复用。系统只使用少量的对象，而这些对象都很相似，状态变化很小，可以实现对象的多次复用。

又被称为蝇量模式，由于享元模式要求能够共享的对象必须是细粒度对象，因此它又称为轻量级模式
